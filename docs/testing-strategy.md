# 测试策略

Scales & Chroma 采用 TDD 开发方式。也就是先写会失败的测试，用测试把目标行为说清楚，再写实现让测试通过，最后再做整理和重构。

这个项目的测试重点不是普通表单或后台 CRUD，而是下面这些核心体验：

- 乐理元素能被组合和编排
- 乐理特征能映射成颜色、粒子、几何和动画
- Canvas 能稳定进行实时渲染
- EXP 和连续练习逻辑可预测、可追踪
- API 返回结构稳定，方便前后端持续联调

## 后端测试

后端测试放在 `backend/app/tests/`。

- `unit/`：单元测试，主要覆盖 EXP 计算、连续签到、数据库模型、Schema 校验、视觉引擎组件等确定性逻辑
- `integration/`：集成测试，后续用于覆盖数据库、服务层和跨模块协作
- `api/`：接口测试，覆盖 FastAPI 路由的输入、输出和错误状态
- `performance/`：性能测试，后续用于高并发、批量数据和实时计算压力场景
- `edge_cases/`：边界测试，后续用于非法日期、极端 BPM、闰年、跨月、跨年等场景

## API 边界测试规划

这部分会作为后续答辩时重点展示的测试设计思路。目标不是只证明“接口能通”，而是证明我们有意识地覆盖了接口最容易出问题的边界。

计划重点覆盖：

- 非法参数：例如负数 `user_id`、空 `topic`、非法日期、空组合、超出限制的 `limit`
- 缺失字段：例如创建练习记录、保存组合、登录注册时的必填字段缺失
- 越权访问：用户 A 不能读取、覆盖、删除用户 B 的练习记录和组合
- 空数据场景：新用户没有练习记录、没有组合、没有解锁特效时接口返回应稳定
- 错误状态码：`400`、`401`、`403`、`404`、`422` 等状态必须和错误原因一致
- 认证基础流程：注册、登录、带用户身份访问自己的数据、访问他人数据被拒绝

后续在测试展示时，可以按下面这个顺序讲：

1. 先测正常路径，锁定 API 契约。
2. 再测非法输入，保证接口不会被脏数据击穿。
3. 再测越权访问，证明最基础的用户隔离成立。
4. 最后测空数据与错误状态，保证系统在“不理想输入”下也可预测。

## 前端测试

前端测试放在 `frontend/src/tests/`。

- `drag/`：沙盘拖拽、编排轨道、积木删除、非法组合、轨道内重排
- `canvas/`：视觉映射和 Canvas 实时渲染器
- `ui/`：用户能看到和操作的界面行为
- `hooks/`：后续用于覆盖 React 状态逻辑
- `responsiveness/`：后续用于覆盖不同屏幕尺寸、UI 重叠、Canvas 缩放

## 认证与用户隔离测试规划

当前项目范围里，认证不会做复杂安全体系，而是做“最基础但可演示”的版本。因此测试重点也会相应收敛：

- 注册成功后能创建独立用户
- 登录后能拿到当前用户身份
- 当前用户只能看到自己的练习记录、组合和成长数据
- 用户不能通过改 `user_id` 访问他人的数据
- 退出后访问受限接口应失败

这样设计的好处是：

- 实现复杂度低，适合当前项目阶段
- 用户隔离目标明确，便于答辩时说明“范围控制”
- 可以把重点资源继续放在沙盘体验和核心闭环上

## 第一批红灯测试

项目一开始先写了两类会失败的测试：

- `backend/app/tests/unit/test_exp_service.py`
- `frontend/src/tests/canvas/visualEngine.test.ts`

它们当时故意引用了还不存在的模块。这样做的目的不是制造错误，而是先把“接下来要实现什么”写成可运行、可追踪的测试目标。

后续每个阶段都尽量保持这个节奏：

```text
写红灯测试 -> 提交并推送 -> 实现功能 -> 跑通测试 -> 再提交并推送
```
